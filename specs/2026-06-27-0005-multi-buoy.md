# 0005 — Multi-buoy + station picker

- **Status:** Accepted
- **Date:** 2026-06-28
- **Relates to:** [0001 — Foundation](2026-06-27-0001-foundation.md) (the "multiple buoys"
  non-goal is lifted), [0002 — Data dictionary](2026-06-27-0002-data-dictionary.md)
  (the schema was already `campaign_id`-keyed and multi-buoy-ready),
  [0004 — Realtime scraper](2026-06-27-0004-realtime-scraper.md) (per-campaign HF layout).

> Adds a **second buoy** — CANDHIS **06402 — Anglet** — and the UI to switch between
> it and 06403. Also adds a top **station bar** (what Olatu is, where the buoys are,
> where the data comes from) and a **map-based buoy picker**. This is the roadmap's
> "multi-buoy (06402 Anglet staged)" item (0001 §11/roadmap).

---

## 1. Why

The Basque coast has two CANDHIS directional Waveriders, ~14 km apart:
**06403 Saint-Jean-de-Luz** (already shipped) and **06402 Anglet** (off La Barre /
the Adour mouth). The owner has the full **06402 archive locally (2009→2026)** in the
exact same CANDHIS dialect as 06403, so the second buoy is a data-ops + UI change, not
a new pipeline. Anglet's archive even reaches back to **2009** (vs 2013 for 06403).

The app was single-buoy by deliberate scope (0001 §1 non-goals), but the data model was
built multi-buoy-ready from day one: `campaign_id` is a column, the HF dataset is laid
out `<campaign>/{raw,data}/…` (0004 §3), and `update.py`/`scrape.py` already take a
`--campaign`. The only thing hard-wired to `06403` was **`build.py`** (via the
`schema.CAMPAIGN_ID` global) and the **webapp data base URL**. This spec finishes the
parameterization and adds the switch UI.

## 2. Buoy identity — 06402 Anglet

| Field | Value |
|-------|-------|
| Station | **06402 — Anglet** |
| Network / operator | CANDHIS, operated by **Cerema** (+ Dept. Pyrénées-Atlantiques 64) |
| Coast | Atlantic / Basque coast, off **Anglet** (Adour estuary / La Barre), ~14 km N of 06403 |
| Position | **43.5322 °N, −1.6150 °W** |
| Sensor | Datawell directional Waverider |
| Cadence | one sea state every **30 minutes** |
| Water depth | not published in open docs → display "not published" |
| Archive span | **2009 → 2026** (identical 73-column CANDHIS schema; same 43 empty cols dropped) |

> **Position provenance:** 43.5322, −1.6150 is the value published by the CANDHIS
> realtime mirrors (allosurf / surf-report, both citing CANDHIS-Cerema). 06403's
> 43.408333, −1.681667 came from Cerema decimal-minutes; if the exact Cerema
> decimal-minute figure for 06402 surfaces, update the single constant in
> `ingest/schema.py` (`BUOYS["06402"]`) and `webapp/src/lib/buoys.ts`.

The realtime feed (`campagne.php?camp=06402`) carries the same 6 columns as 06403,
**including `TempMer`** — so 06402 sea-temperature history likewise accumulates forward
from the scraper's first run (0001 §2.4).

## 3. Pipeline — finish the campaign parameterization

The schema's `ARCH_MAP`/`REEL_MAP`/`UNITS`/`HEADLINE`/sentinel are CANDHIS-wide and stay
shared. Only the **per-buoy identity** is keyed by campaign:

- `ingest/schema.py`: replace the single `BUOY` dict with a **`BUOYS` registry**
  (`{campaign_id: {…identity…}}`) + a `buoy(campaign)` accessor. `CAMPAIGN_ID` stays as
  the **default** campaign (06403) so existing call-sites and CLIs keep working.
- `ingest/build.py`: `build(src, out, campaign=CAMPAIGN_ID)` — thread `campaign` through
  `read_archive`/`read_realtime` (glob `Candhis_<campaign>_*`) and into the manifest
  (`buoy(campaign)`, `campaign_id`, `year/<campaign>_YYYY.parquet`). No more reliance on
  the module global for file names/identity.
- `ingest/scrape.py`: already `--campaign`-aware; make the per-run lock file
  campaign-scoped (`.scrape_<campaign>.lock`) so 06402 and 06403 scrapes don't block
  each other.
- `ingest/update.py`: pass `campaign` into `build_mod.build(...)`.

**Seeding 06402:** one-time, same path as 06403 —
`pixi run update --campaign 06402 --seed-src /Users/hadim/Data/olatu/06402` — uploads
`06402/raw/*` (archive + reel) and the freshly built `06402/data/*` tiers to the dataset.

## 4. CI

`refresh-data.yml` refreshes **both** campaigns every 30 min. A matrix over
`['06402','06403']` (one job per campaign, each with its own archive cache key) keeps the
jobs independent and the logs readable; the `concurrency` group still serializes against
itself. The archive cache path becomes per-campaign (`hfdata/<campaign>/raw`).

## 5. Webapp

### 5.1 Data layer (campaign-aware)
- The data base stops baking in `06403`. New shape: a dataset **root**
  (`…/resolve/main/`) + `<campaign>/data/` appended at call time. `VITE_DATA_BASE_URL`
  now overrides the **root** (must end in `/`).
- `loadManifest/loadLatest/loadRecent(campaign)` and
  `loadParquetTier(name, columns, campaign)` all take the campaign.
- A small static **`buoys.ts` registry** (`campaign_id`, `name`, `short`, `lat`, `lon`,
  blurb keys) lets the picker + locator map render **before** any manifest loads (instant
  first paint, and resilient if one buoy's data is briefly unavailable). The selected
  buoy's manifest still drives its detailed station facts. The registry's lat/lon must
  match each buoy's manifest.

### 5.2 Switch UX
- **Selected campaign** is React state, persisted to `localStorage` (`olatu.campaign`),
  default **06403**. Switching reloads that campaign's manifest/latest/recent + resets the
  charts' detail caches; the banner/charts re-render for the new buoy. Theme, locale and
  the live-refresh poller are unaffected.
- Two affordances, kept in sync:
  - a **segmented control** (accessible, instant, always works) in the station bar;
  - a **map locator** showing **both** buoys as markers — the active one highlighted,
    click an inactive marker to switch. This is the "buoy selector on a map" the owner
    asked for. It is **lazy** (MapLibre loads after first paint behind a themed
    placeholder) so it never taxes the banner-first paint (0001 §7.3). The per-buoy
    **detailed** map (ExpandedMap) is unchanged and now follows the selected buoy.

### 5.3 Station bar (top)
A new top section above the conditions banner that **separates the app frame from the
data**: the **Olatu** wordmark + a one-line description, a short "two CANDHIS buoys on the
Basque coast" line naming both stations, and a **data-source** line linking the live
source (**CANDHIS**) and the **open dataset on Hugging Face** (`hadim/olatu`). The
identity header keeps the brand; the selected buoy's name now lives in the conditions /
station context rather than as the page `<h1>` (there are two buoys now).

### 5.4 Data-source surfacing (also a README ask)
Both the app (station bar + footer) and the README state plainly that the tiers are the
**open HF dataset** [`hadim/olatu`](https://huggingface.co/datasets/hadim/olatu),
refreshed every 30 min from CANDHIS, CORS-served and reusable by anyone.

## 6. Out of scope (deliberately deferred)
- Comparing two buoys **side by side** (overlaid charts). One-at-a-time switch only.
- A buoy-count beyond these two; the registry + pipeline scale, but no third buoy is
  staged.
- Per-buoy glossary differences (the variable set is identical across CANDHIS buoys).

## 7. Decisions
1. ✅ **Add 06402 Anglet**, archive 2009→2026, seeded from the local CSVs.
2. ✅ **Switch, don't compare** (one buoy shown at a time) — simplest UX that meets the ask.
3. ✅ **Map picker is lazy** (static placeholder → MapLibre) to protect first paint; a
   segmented control is the always-instant fallback and primary a11y control.
4. ✅ **Default buoy stays 06403** (the original), persisted choice overrides.

## Revision — 2026-06-28 (Cap Ferret + URL deep-link)

**+ 03302 Cap Ferret (realtime-only).** A third buoy, off the Gironde / Arcachon coast
(**44.6525 °N, −1.44667 °W**). No historical archive was available locally, so it is
seeded **realtime-only**: the scraper grows its reel and `build.py` builds purely from
that tail — its history **accumulates forward** from the first scrape (exactly like sea
temperature, 0001 §2.4). Pipeline change: `read_archive` returns `None` (instead of
raising) when there is no `*_arch.csv`, and `assemble` accepts either feed absent; a
build aborts only if **both** archive and realtime are missing. Drop archive CSVs into
the campaign's `raw/` later and they coalesce in to backfill history.

The CI matrix (§4) now covers `['06403','06402','03302']`. The frontend registry +
locator map are unchanged in shape; to stay readable with buoys far apart (Cap Ferret is
~140 km north of the two Basque buoys, which are only ~14 km apart) the **locator shows
each marker's name only when active or hovered** — inactive markers are dots.

> **Probed but not added:** **01705 Royan** is live but its realtime table has 7 columns
> (no `TempMer`), which the scraper's 8-column assertion rejects — deferred. **01704
> Oléron**'s feed is frozen at 2025-05-17 (effectively offline) — nothing live to show.
> Both can be added later (Royan needs a 7-column scraper branch; Oléron needs archive
> CSVs or to come back online).

**URL deep-link (`?buoy=<id>`).** The selected buoy is now also encoded in the URL for
easy sharing: `…/?buoy=06402` opens that buoy. Precedence on load: **URL > persisted
(`olatu.campaign`) > default 06403**; on switch the app `replaceState`s the new
`?buoy=` (no history spam, back button leaves the site normally) and persists it.
(`pushState` + a `popstate` handler was tried first but caused spurious re-selection in
messy history states — `replaceState`-only is the robust choice and still fully
shareable.)
