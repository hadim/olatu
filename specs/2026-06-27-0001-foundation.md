## 1. Vision

A **beautiful, fast, fully-static web app** that lets anyone — surfer, sailor,
scientist, or curious local — read what the sea is doing at the
**Saint-Jean-de-Luz wave buoy** right now, and explore its full history with no
friction.

Design north stars:

1. **You open it and instantly know what you're looking at.** A signature
   *current-conditions* readout answers "how big, from where, how warm" in under
   two seconds, before reading a single axis.
2. **Pixel-perfect, branded data-viz** — not a generic charting-library look.
3. **Effortless time travel** — today / yesterday / this week / last week / this
   month / this year / any past year, plus a precise date-range cherry-picker.
4. **Every value is explained** — a complete but plain-language definition for each
   variable, one tap away, never cluttering the screen.
5. **Desktop and mobile are both first-class.**
6. **Multilingual** — English, French, Spanish at minimum.

Non-goals (for now): multiple buoys, forecasting/modelling, user accounts, a
backend/API of our own.

## 2. The data — what it actually is

> Profiled directly from the real CANDHIS export the owner downloaded
> (`/Users/hadim/Data/fff`, CSV, 2013→2026). These numbers supersede anything the
> old synthetic `data/*.parquet` files implied.

### 2.1 Source & buoy

- **Network:** CANDHIS — the French national in-situ wave-measurement observatory,
  operated by **Cerema** (with the Pyrénées-Atlantiques department).
- **Buoy:** campaign **06403 — Saint-Jean-de-Luz**, Atlantic / Basque coast (the
  inner southern Bay of Biscay), ~3 km off the **Belharra** reef.
  - Position: **43.408333 °N, −1.681667 °W**.
  - Sensor: Datawell directional Waverider; **one sea state every 30 minutes**.
  - Water depth: not in public docs (behind a keyed API) → display "not published".
- Full identity & sources: see [0002](2026-06-27-0002-data-dictionary.md).

### 2.2 Two file shapes (both `;`-delimited, half-hourly)

| Feed | File | Rows | Columns | Sea temp? |
|------|------|------|---------|-----------|
| **Archive** | `Candhis_06403_YYYY_arch.csv` (2013→2026) | **214,811** total | 73 (only ~30 populated) | **No** |
| **Realtime** | `Candhis_06403_YYYY-MM-DD_reel.csv` (rolling 48 h) | ~97 | 7 | **Yes** (`TempMer`) |

### 2.3 Critical findings (these drive the architecture)

1. **`999.999` is the missing-value sentinel** (verified — earlier "1000" readings
   were just `%.2f` rounding of `999.999`). It appears uniformly across `HM0`, `TP`,
   `THETAP`, `SIGMAP`, `T02`, `TE`, `H13D`, `HMAXD`… A 999.999 m wave / s period /
   ° direction is impossible. **The ingest nulls any value ≥ 999.99** (threshold, not
   exact equality, to be float-safe — no real variable here reaches it; crucially
   this preserves legitimate `peak_direction_deg` values up to 360). Spectral-derived
   fields carry sentinels more often than the time-domain ones (`H13D`, `HMAXD`).
2. **43 of 73 archive columns are 100 % empty for this buoy:** `QUALITE`, `NBSYS`,
   and the entire **S1–S4 spectral-partition block** (40 cols). The "up to 4 wave
   systems" partitioning was never populated for 06403. **We drop all 43.** The
   *usable* schema is **~30 columns**. (Definitions are kept in 0002 as reference,
   marked "not available for 06403".)
3. **Real outages exist:** **93.9 % coverage**, **66 gaps > 1 h**, the largest
   **50 days** (May–Jul 2018), several multi-week. Charts must **break the line
   across gaps** (no interpolation), and history presets can legitimately land on
   empty stretches → the UI needs a graceful "no data for this period" state.
4. **Sea temperature exists only in the realtime feed**, never in the archive.
5. **The archive lags ~6 weeks** (last archive row 2026-05-13; realtime fills to
   "now"). The two feeds must be merged into one canonical series.

### 2.4 The store is append-only and grows (owner's intent)

The future CI scrapes the realtime feed every ~30 min and **appends** new rows.
Consequence: **sea-temperature history accumulates forward** from whenever the
scraper starts — null before that, populated after. So the data model is
**"immutable backfill (2013→cutoff) + a live-growing tail"**, not a static dump.
This is a first-class design input (see §6 storage strategy, and the temperature
empty-state handling in §7).

### 2.5 Headline variables (the ~8 worth surfacing prominently)

`significant_wave_height_m` (H1/3 — the hero) · `max_wave_height_m` (Hmax) ·
`peak_period_s` (Tp) / `significant_period_s` (Th1/3) · `peak_direction_deg` ·
`peak_directional_spread_deg` · `sea_temperature_c` · plus the
`spectral_significant_height_hm0_m` for the "scientific" view.

## 3. Architecture overview

```
                 ┌──────────────────────────────────────────┐
   CSV / scrape  │  Python (polars) ingest  — run by CI       │
   ────────────► │  • clean (sentinel→null), normalize schema │
                 │  • merge archive + realtime tail           │
                 │  • emit TIERED static files (JSON+Parquet)  │
                 └───────────────────┬──────────────────────┘
                                     │ committed to deploy output (GH Pages)
                                     ▼
   ┌───────────────────────────────────────────────────────────────┐
   │  Static SPA (React + TS, Vite)  — GitHub Pages, no backend     │
   │  • hyparquet reads Parquet in the browser (range + projection) │
   │  • uPlot canvas charts + custom direction/rose layer           │
   │  • MapLibre (lazy) for the expandable map                      │
   │  • Paraglide i18n (EN/FR/ES)                                   │
   └───────────────────────────────────────────────────────────────┘
```

Everything is static. No server we run. The only "compute" is the scheduled CI job
that regenerates data files.

## 4. Recommended stack

Derived from 8 parallel research tracks, reconciled into one coherent stack, then
adversarially stress-tested. Where the pre-mortem pushed back, the **decision below
reflects the post-critique call.**

| Area | Decision | Why (one line) |
|------|----------|----------------|
| **Framework** | **React 19 + Vite 7**, migrate to **TypeScript** | Keep the existing shell; deepest ecosystem for the hard parts (date-picker, dialog, popover); all other libs are framework-agnostic. |
| **Language** | **TypeScript** (`.jsx`→`.tsx`) | Type-safe contracts for the canonical schema, the loader state machine, and i18n keys. Zero runtime cost on a static site. |
| **Styling** | **Tailwind CSS v4** (`@tailwindcss/vite`, CSS-first `@theme` tokens) | Cleanest way to express the dual-theme token system; native substrate of shadcn/ui. |
| **UI primitives** | **shadcn/ui** (copy-in) on the **Radix** backend | Own the component source (ideal for a solo static project); Radix's larger ecosystem is the lower-risk pick for the date-picker/dialog/popover/segmented controls. *(Base UI noted as a future option, not adopted now.)* |
| **Charts (quantitative)** | **uPlot 1.6.x** in a small custom React hook | **The decisive call.** Canvas 2D → total pixel control (the owner's #1 demand, the antithesis of Plotly); native synced multi-panel; ~50 KB; 60 fps over the whole archive on mobile. |
| **Charts (directional + rose)** | Thin custom Canvas layer using `d3-shape`/`d3-scale`/`d3-array` via uPlot draw hooks | No permissive lib renders wave-direction arrow glyphs on a time axis natively — it's custom work *everywhere*; uPlot makes it cheapest and fixes the 0/360° wrap. Powers the signature **compass-rose swell dial**. |
| **Map (expanded)** | **MapLibre GL JS**, lazy-imported; keyless vector tiles | Only path meeting "no API key / no billing on GH Pages." Default tile source: **OpenFreeMap** (keyless hosted) *or* self-hosted **PMTiles** committed to the deploy output (fully independent). Dark nautical theme. |
| **Map (mini preview)** | Static committed **PNG** thumbnail + CSS marker dot | Avoids loading MapLibre's ~200 KB WebGL runtime on first paint; instant; heavy stack loads only on "expand". |
| **i18n** | **inlang Paraglide JS v2** + React adapter + native **Intl** | Compile-time, tree-shaken, type-safe; localStorage/header detection (no URL routing → no GH Pages base-path pain). |
| **Glossary content** | Per-locale **dynamic-imported JSON** (`glossary.{en,fr,es}.json`) | 30+ vars × 2 levels × 3 langs is too heavy for inline messages; code-split keeps it translator-friendly and downloads only the active language. |
| **Data loading** | **hyparquet** (zero-dep, no WASM) + tiered files | Correct tool at this scale. DuckDB-WASM / parquet-wasm **rejected** (multi-MB payloads; GH Pages can't set COOP/COEP for SharedArrayBuffer). |
| **Python ingest** | **polars** CSV→Parquet, run as **pixi tasks** (no installable package) | Minimal, fast; produces the tiered file set, does sentinel-cleaning + temperature merge + server-side downsampling. |

### 4.1 Rejected, for the record

- **Plotly.js** (incumbent) — ~3.6 MB, generic look, renders direction as a
  wrapping degree-line. Removed entirely.
- **ECharts** — strong runner-up (one lib for charts *and* geo map) but ~1 MB and
  still reads as "an ECharts chart"; its map advantage is moot since we need
  MapLibre for keyless tiles anyway. **Kept as the documented fallback** if the
  uPlot interaction glue proves too costly.
- **Highcharts** (only lib with native wind-barbs) — commercial license.
- **DuckDB-WASM / parquet-wasm** — multi-MB + SharedArrayBuffer constraints.
- **Pandas / matplotlib / seaborn / s3fs / cuda** — removed from the Python side.

## 5. Data pipeline & storage strategy

### 5.1 polars ingest → tiered static files

The ingest emits this file set (served from the deploy output; see §5.3 for *where*
they live so we don't bloat git):

1. **`manifest.json`** (~1 KB) — buoy metadata, the variable dictionary (canonical
   name, unit, i18n keys), full date span, the list of year files **with
   `byteLength`** (so hyparquet skips the HEAD request), per-variable
   **coverage ranges** (esp. `sea_temperature` coverage → drives empty-state UI),
   and `generated_at`. Loaded first; everything else discovered from it.
2. **`latest.json`** (~5–15 KB) — last 48 h @ 30 min, realtime-derived, **including
   sea temperature**. Powers the current-conditions banner + swell dial + default
   landing chart. Eager.
3. **`recent.json`** (~30–60 KB) — last 30 days @ 30 min, archive+realtime merged.
   Prefetched after first paint → today / 24 h / 7 d / this-month presets with zero
   extra round-trips.
4. **`year/06403_YYYY.parquet`** — one per year, full canonical schema @ 30 min,
   **Snappy**, **`row_group_size ≈ 1440`** (≈ monthly → many row groups). Lazy,
   column-projected. Past years are immutable; only the current year changes.
5. **`hourly.parquet`** — hourly means of headline vars, for multi-month / 1–3 yr
   views. *(Capped to a recent window or split per-year if the full-archive hourly
   table gets large.)*
6. **`daily.parquet`** — daily means of headline vars for the full-archive / heat-
   ribbon overview (~5 k rows over 13 yr → loads instantly, plotted in full).

**Hard ingest rules (CI-asserted):**
- Sentinel `999.999` (null any value ≥ 999.99) → **null**.
- Drop the 43 empty columns.
- Write **Snappy** (so the browser never needs `hyparquet-compressors`) and assert
  **`num_row_groups > 1`** per Parquet (the current single-row-group files defeat
  range requests — this is a real bug we're fixing).
- Timestamps written **UTC-aware**; frontend renders in **Europe/Paris** (fixes the
  old `new Date(value)` browser-local drift).
- Temperature merge is deterministic: realtime is the sole temp source; prefer
  realtime rows for the last 48 h, archive for everything older → exactly one
  canonical row per half-hour.

### 5.2 Loading state machine (frontend)

Boot: fetch `manifest.json` + `latest.json` → banner + default view paint
immediately → prefetch `recent.json`. On a range request, pick the **smallest
covering tier**: ≤30 d → `recent.json` (in memory) · ≤~3 mo → year parquet(s) via
hyparquet + column projection · single year → year or `hourly` · multi-year/all →
`daily`. Cache loaded tiers in memory; **persist parsed years in IndexedDB** (GH
Pages caps cache at `max-age=600`, so this is do-it, not optional). Client-side
**LTTB** is a *last-mile cosmetic pass only* — never used to decimate data we had to
fully download anyway.

### 5.3 ⚠️ Git-churn strategy (the #1 long-term risk — needs a decision)

Committing data every 30 min to `main` would **kill the repo within ~1 year**
(Snappy Parquet is incompressible binary → a near-full new blob per commit,
unprunable without history rewrite; projection ~hundreds of MB–GBs/yr). So:

**Separate immutable history from the churning tail:**
- **Immutable backfill (2013 → cutoff):** written **once**, never rewritten → safe
  to keep in `main` (no churn) or on the deploy branch. ~a few MB total.
- **Churning tail + rebuilt tiers (`latest/recent/manifest`/current-year):**
  rewritten every 30 min → must **NOT** accumulate in `main`'s history.

**Decision (2026-06-27): commit data to `main` for now.** The owner chose the
simplest path to get moving: the ingest writes the tiers into the repo and they are
committed to `main`. **This is acknowledged technical debt** — at ~48 commits/day of
incompressible Snappy blobs it trends toward hundreds of MB/yr in `.git` and will
eventually need addressing.

**Documented migration path when it bites** (no code rewrite, just CI topology):
publish the built site **and** data tiers to the **`gh-pages` branch with
`force_orphan: true`** (single commit, no history); keep the immutable backfill in
`main` (write-once → no churn) and move only the 30-min churning tail to a
force-pushed orphan data branch (or Release assets). **Never use Git LFS** here (its
quotas are exhausted in weeks and LFS objects are also never pruned). Until then,
keep the ingest output path configurable so this switch is a CI change only.

## 6. Feature spec

### 6.1 Current-conditions banner (the signature)
- The hero: **H1/3 significant wave height** as a big readout inside a **compass-
  rose swell dial**; a bold arrow shows the direction the swell comes **from**
  (labelled, e.g. "from WNW 292°"), colored by a cyclical hue scale; ring fill/glow
  encodes height.
- Secondary gauges: period, direction (compass text), spread (as a ± wedge),
  Hmax, each with a 12 h sparkline.
- **Sea-surface temperature** in its own warm-accent cell (big °C + today min/max).
- **Staleness** (nearest sample, ≤6 h target): fresh ≤2 h (teal, slow pulse) /
  aging 2–6 h (amber) / stale >6 h (desaturate everything, dotted ring, explicit
  "last reading from HH:MM, N h ago"). Always show the real timestamp.

### 6.2 Synced multi-panel charts
- Stacked panels sharing **one x-axis**, unified crosshair + single value card:
  (1) wave height H1/3 + Hmax envelope with a faint sequential-scale background
  band; (2) period; (3) **direction as colored glyphs** on an N/E/S/W axis with
  spread as a translucent band (not a second line); (4) sea temperature.
- A custom **navigator / brush strip** replaces the Plotly rangeslider.
- Gaps render as breaks; missing-temperature spans render as a hatched "no data"
  band or a collapsed panel (driven by `manifest` coverage ranges), never a line to
  zero or an empty pane.

### 6.3 History navigation
- **Preset chips** — relative row (Today / Yesterday / 24 h / This week / Last week
  / This month / This year) + a scrollable **per-year** row (2013…2026).
- **Date cherry-picker** — dual-month calendar (desktop popover / mobile full-screen
  sheet); days with data subtly marked, big-swell days flagged.
- **Heat-ribbon timeline** — a full-width ribbon colored by the wave-height scale
  across the whole record (see the storms at a glance), with draggable focus
  handles. All three mechanisms stay in sync.

### 6.4 Map
- Mini static thumbnail + marker; tap → MapLibre dialog (dark nautical theme,
  pulsing marker, attribution). Optional OpenSeaMap seamark overlay (best-effort).

### 6.5 Glossary (3-tier progressive disclosure)
- **Tier 1** — labels that already teach ("from WNW 292°", "spread ±18° (clean)").
- **Tier 2** — a quiet "i" per metric → compact popover (name, one-line def, unit,
  "why it matters").
- **Tier 3** — a "Definitions" slide-over listing every variable with full defs,
  the original CANDHIS French term, typical ranges, and a sea-state reference scale
  that doubles as the chart legend.

## 7. UX / UI direction — "Station Reading"

A **premium marine instrument** crossed with a **scientific buoy station page** and
the at-a-glance clarity of a **surf forecast**. The screen reads like the buoy is
"on watch" and reporting in. Anti-template guardrails: no Material/Bootstrap card
grid, no generic `#2563eb` blue, no Plotly rainbow. Instrument glass over flat
cards; data-ink over chrome; one luminous accent.

### 7.1 Color (hue = direction, luminance/saturation = intensity)
- **Dark (default — "night watch glass"):** base `#0A1622`, raised `#0F2233`,
  hairline `#1E3A4F`, text `#E8F1F5`/`#8FA9B8`, **brand accent `#38E1C6`**
  (instrument cyan-teal), warm/temp `#FF8A5B`, danger `#FF5C7A`.
- **Light ("deck in daylight"):** base `#EEF4F6`, raised `#FFFFFF`, brand `#0E8C7A`,
  warm `#E85C2B`, text `#0B2233`.
- **Wave-height sequential scale** (hero, dial fill, heat-ribbon): `#BFE9E0`
  (glassy) → `#6FD3C4` → `#38B8C9` → `#2E8FC4` → `#3D5FBE` → `#7A4FC0` → `#B83D8E`
  (storm).
- **Direction cyclical scale** (0–360° is circular → a hue wheel, in OKLCH so 359°
  sits next to 1°): N `#38E1C6`, E `#7CC4FF`, S `#FFC857`, W `#FF7AA8`.

### 7.2 Typography (all free / OFL, self-hosted)
- **Display / readout:** Space Grotesk (hero numbers, eyebrows; tabular figures).
- **Body / UI:** Inter.
- **Data / ticks / timestamps:** IBM Plex Mono (the "station page" tell).

### 7.3 Layout
- **Desktop:** single-column station report — sticky identity header → full-width
  banner → mini map + station-facts → 4 synced chart wells → sticky time-range
  controls + heat-ribbon → footer. ~1280 px max, 8 px vertical rhythm.
- **Mobile:** card stack, most-glanceable first — dial goes full-width, gauges
  become a 3-up tile row, charts get taller with **tap-to-inspect** (value card
  pinned at top, not under the finger) + **pinch-zoom** + a "Reset" pill; presets
  become a snap-scrolling row; picker is a bottom sheet; all touch targets ≥44 px.

References absorbed: Surfline, Windy, NOAA NDBC / CDIP, B&G/Garmin chartplotters,
Surf-Forecast, Apple Weather/Tide, CARTO dark-matter.

## 8. i18n plan

- **UI strings:** Paraglide compile-time messages, `en`/`fr`/`es`, base `en`.
  Detection `["localStorage","preferredLanguage","baseLocale"]`; switch via a header
  segmented control using `setLocale(loc, {reload:false})` so range/zoom state
  survives.
- **Glossary:** per-locale dynamic-imported JSON, **keyed off the canonical schema**
  (one source of truth → manifest dictionary + glossary files + a **CI key-parity
  check** that *fails* the build on drift). Runtime fallback chain `es→en`, `fr→en`
  so a partial locale degrades to English, never an empty popover.
- **Formatting:** native `Intl` (`NumberFormat` / `DateTimeFormat` /
  `RelativeTimeFormat` for "updated N h ago"); units come from the schema, not
  hardcoded per string.
- **Staging:** EN + FR are launch-critical (FR = the data's native language); ES may
  trail by one phase if needed — but all three are in scope.
